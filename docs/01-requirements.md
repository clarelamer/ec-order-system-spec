## 1. 背景與問題

電商平台的訂單流程是「錢與貨的交會點」，也是最容易出錯、最需要嚴謹狀態管理的環節。一張訂單從成立到結束，會經過付款、出貨、收貨等多個階段，任何一步都可能岔出非預期路徑：付款逾時、庫存不足、使用者取消、部分退貨。這些「不快樂路徑」若沒有明確定義，就會變成客訴與帳務糾紛的來源。

本專案交付一套訂單流程的規格與 PoC，目標是把訂單生命週期的**每個狀態與轉移條件**定義清楚，讓工程、客服、財務對「訂單現在處於什麼狀態、接下來能做什麼」有單一的事實來源。

## 2. 角色（Actors）

| 角色 | 說明 | 在訂單流程中的關注點 |
|------|------|--------------------|
| 買家（Customer） | 已註冊會員，下單者 | 下單、付款、查看訂單狀態、取消、申請退貨 |
| 系統（System） | 訂單服務本身 | 庫存校驗、狀態轉移、付款逾時自動處理 |
| 金流服務（Payment Gateway） | 外部付款服務（本專案以 mock 表示） | 回傳付款成功／失敗的 webhook |
| 營運人員（Operator） | 後台人員 | 確認出貨、處理退貨審核 |

> 範圍說明：營運人員與金流服務的**內部系統不在本專案範圍**，只定義它們與訂單服務的互動介面（狀態轉移的觸發點）。

## 3. 範圍界定

### 3.1 涵蓋（In Scope）

- 訂單成立（含下單時的庫存校驗與金額計算）
- 付款流程（含付款逾時自動取消）
- 出貨與完成
- 取消（付款前使用者主動取消、付款後協商取消）
- 退貨（全額退貨與**部分退貨**）
- 訂單、訂單明細、商品、會員、付款的資料關聯

### 3.2 不涵蓋（Out of Scope）

- 購物車與結帳 UI（假設下單 request 已帶齊商品與數量）
- 商品上下架與庫存補貨後台
- 金流商、物流商的實際 API 串接（以 mock/webhook 介面表示）
- 發票、優惠券、紅利點數（在資料模型預留擴充位，但不實作邏輯）
- 多倉庫、多幣別

## 4. User Story

以 `身為<角色>，我想要<目標>，以便<價值>` 格式撰寫，並附**驗收條件（AC）**。驗收條件是下游規格（狀態機、API）的直接依據。

### US-01 建立訂單

> 身為**買家**，我想要用選好的商品與數量建立一張訂單，以便鎖定價格並進入付款。

**驗收條件：**
- AC-1：下單時系統校驗每項商品的庫存，任一項不足則整張訂單建立失敗，回傳不足的品項
- AC-2：訂單金額 = Σ（品項單價 × 數量），單價以**下單當下**的商品價格快照為準，日後商品改價不影響已成立訂單
- AC-3：建立成功後訂單狀態為「待付款」，並為每項商品**預扣庫存**（reserve），避免超賣
- AC-4：同一筆請求重複提交（相同 idempotency key）只會建立一張訂單

### US-02 付款

> 身為**買家**，我想要對待付款訂單完成付款，以便讓賣家出貨。

**驗收條件：**
- AC-1：只有「待付款」狀態的訂單可付款，其他狀態回傳 409 衝突
- AC-2：付款成功後，訂單狀態轉為「已付款」，預扣的庫存轉為正式扣減
- AC-3：付款失敗不改變訂單狀態，訂單仍為「待付款」，可重試
- AC-4：訂單建立後 **30 分鐘內未付款**，系統自動將訂單轉為「已取消」並釋放預扣庫存

### US-03 出貨與完成

> 身為**營運人員**，我想要對已付款訂單標記出貨，以便買家收貨後完成交易。

**驗收條件：**
- AC-1：只有「已付款」訂單可出貨，出貨後狀態轉為「已出貨」並記錄出貨時間
- AC-2：「已出貨」訂單由買家確認收貨或系統在出貨後 N 天自動轉為「已完成」
- AC-3：「已完成」為終態，不可再轉移（退貨走獨立流程）

### US-04 取消訂單

> 身為**買家**，我想要在出貨前取消訂單，以便在改變主意時不損失金錢。

**驗收條件：**
- AC-1：「待付款」訂單可直接取消，釋放預扣庫存，狀態轉為「已取消」
- AC-2：「已付款」但未出貨的訂單，取消需觸發**退款**，退款完成後轉為「已取消」
- AC-3：「已出貨」與「已完成」訂單不可取消，只能走退貨流程
- AC-4：「已取消」為終態

### US-05 退貨（含部分退貨）

> 身為**買家**，我想要對已完成訂單的部分或全部品項申請退貨，以便退回不滿意的商品。

**驗收條件：**
- AC-1：只有「已出貨」或「已完成」訂單可申請退貨
- AC-2：退貨可**指定品項與數量**（部分退貨），退款金額 = Σ（退貨品項單價 × 退貨數量）
- AC-3：退貨經營運人員審核通過後，退款並回補對應庫存
- AC-4：全部品項退貨完成後訂單轉為「已退貨」；部分退貨完成後訂單維持「已完成」，但記錄退貨明細
- AC-5：同一品項的累計退貨數量不可超過原購買數量

## 5. 關鍵業務規則（Business Rules，跨 Story）

| 編號 | 規則 | 影響的下游 |
|------|------|-----------|
| BR-1 | 價格快照：訂單明細記錄下單當下單價，不隨商品改價變動 | ER：`order_item.unit_price`；API：建單 response |
| BR-2 | 庫存兩階段：下單預扣（reserve）→ 付款正式扣減；取消/逾時釋放 | 狀態機：狀態轉移的 side effect |
| BR-3 | 付款逾時 30 分鐘自動取消 | 狀態機：`待付款 --逾時--> 已取消` |
| BR-4 | 冪等性：建單與付款以 idempotency key 防重複 | API：request header |
| BR-5 | 部分退貨累計不超過原數量 | ER：`return_item`；API：退貨端點驗證 |
| BR-6 | 金額一律以最小貨幣單位（分）儲存為整數，避免浮點誤差 | ER：金額欄位型別；PoC：Prisma schema |

## 6. 非功能性需求（NFR，PoC 範圍內示意）

- **一致性**：庫存扣減與狀態轉移須在同一交易（transaction）內，避免中間狀態
- **可觀測性**：每次狀態轉移寫入 `order_status_history`，可追溯訂單完整生命週期
- **冪等性**：對外的寫入端點（建單、付款）支援 idempotency key

## 7. 待釐清問題（Open Questions）

> 真實專案中這些會回到需求訪談。此處先以**預設決策**推進，並標注，方便日後討論。

- Q1：出貨後自動完成的天數 N？→ 預設 **7 天**
- Q2：部分退貨後，運費是否退？→ 預設**不退運費**，退貨僅退商品金額
- Q3：付款逾時時間是否隨商品類型不同？→ 預設**全站統一 30 分鐘**
